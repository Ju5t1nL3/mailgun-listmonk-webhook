## Context

### Problem

Listmonk doesn't natively support bounce webhooks for **Mailgun**.

Despite community interest (see [Listmonk Issue #2032](https://github.com/knadh/listmonk/issues/2032)), it was eventually marked as stale, then eventually closed as "not planned".

### Solution

This microservice sits between Mailgun and Listmonk, verifying and translating Mailgun's payload to Listmonk's expected format.

---

## Prerequisites

You must have:
1. A running instance of **Listmonk** (accessible via HTTP/HTTPS) with the Mailgun SMTP already set up.
2. A **Mailgun** account with an active sending domain
3. Your Mailgun **HTTP Webhook Signing Key** (found in Mailgun Settings > API Security)

## Configuration

### Environment Variables

| Variable | Type | Required | Description |
| - | - | - | - |
| `ENVIRONMENT` | Environment ("development" or "production") | No | Default: `"development"`. Specify if you are running code locally of for production. Hides the docs routes if in production |
| `MAILGUN_SIGNING_KEY` | string | Yes | Found in Mailgun Dashboard > Settings > API Security. |
| `LISTMONK_URL` | string | Yes | The base URL of your Listmonk instance (e.g., `https://listmonk.yourdomain.com`). |
| `LISTMONK_API_USER` | string | Yes | Your Listmonk administrator username. |
| `Listmonk_API_TOKEN` | string | Yes | Your Listmonk administrator password. |
| `REQUIRE_LISTMONK_TAG` | boolean | No | Default: `False`. If `True`, ignores webhooks unless they contain the `listmonk` tag. Useful if you use one Mailgun domain for multiple apps. More about it [here](#enforcing-the-listmonk-tag-(require_listmonk_tag%3Dtrue)). |
| `ENABLE_CAMPAIGN_TRACKING` | boolean | No | Default: `False`. If `True`, extracts the `campaign_uuid` from mailgun payload to map bounces to specific Listmonk campaigns. |

### Advanced Setup

#### Enforcing the Listmonk Tag (`REQUIRE_LISTMONK_TAG=true`)
If you use the same Mailgun sending domain for multiple applications, you want to ensure that this webhook **only** processes generated by Listmonk.

To make this work, first set the `REQUIRE_LISTMONK_TAG` environment variable to `True`. Then, you must tell Listmonk to attach a Mailgun tag to every email it sends:
1. Log in to your Listmonk Admin Dashboard.
2. Navigate to **Settings > Settings > SMTP**.
3. Scroll down to the **Custom headers** field and add the following:
```json
[{"X-Mailgun-Tag": "listmonk"}]
```

## Set it Up!

### Deployment/Installation

#### Pre-built Docker Image (Recommended)
The fastest way to deploy is using the officially published image from the GitHub Container Registry (GHCR).

1. Go to your hosting dashboard (e.g., Railway, Render)
2. Select **Deploy from Docker Image**.
3. Enter the registry URL for this project:

```bash
ghcr.io/Ju5t1nL3/mailgun-listmonk-webhook:latest
```

4. Add the your chosen [Environment Variables](#environment-variables) and deploy!

#### Build from GitHub
If you plan to [modify the code](#local-development), or just prefer to build from scratch, you can deploy from the repository.

1. Fork or clone this repository to your own GitHub account.
2. Select **Deploy from GitHub repo** and choose your repository
3. Add your [Environment Variables](#environment-variables) and deploy!


### Dashboard Setup
Once your microservice is deployed and accessible via a public URL, you need to configure Mailgun and Listmonk

#### Listmonk Setup

1. Log in to your Listmonk Admin Dashboard
2. Navigate to **Settings > Settings > Bounces**
3. Enable Bounce Processing
    - Adjust bounce counts and actions as you please
        - **NOTE**: Complaints are treated as hard bounces in this microservice. Unfortunately, the Listmonk Webhook API has no way to classify complaints.
4. Enable Bounce Webhooks
5. Save settings

#### Mailgun Setup

1. Log into your Mailgun Dashboard
2. Navigate to **Send > Webhooks.**
3. Click Add webhook
    - I highly recommend using a **domain-level webhook** specifically for your sending domain
    - However, if this is the only application using your mailgun account, you can use an **account-level webhook**. But, domain-level prevents your service from accidentally processing events from your other unrelated apps.
4. **HTTP post URL**: Enter your deployed service  URL followed by `/webhook` (e.g., `https://webhook.yourdomain.com/webhook`)
5. Select the following events:
    - **Permanent Fail** (Hard Bounces)
    - **Temporary Fail** (Soft Bounces)
    - **Spam Complaints** (Treated as Hard Bounces in this microservice)
6. Save the webhook.

## Local Development

1. If `uv` is not installed already, you can use the following command on macOS and Linux:
```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```
or read more about it [here](https://docs.astral.sh/uv/getting-started/installation/)

2. Sync dependencies: `uv sync`

3. Run the server
    - If you don't want to manage a virtual environment, simply run `uv run uvicorn app.main:app --reload`
    - If you don't want to write `uv run` every time, you can activate the venv (e.g., `source .venv/bin/activate` on macOS) then run `uvicorn app.main:app --reload`

4. You can run tests locally with `uv run pytest tests/`

5. You can lint using `uv run ruff check`
6. You can type check using `uv run pyright app tests`
7. You can format using `uv run ruff format`
